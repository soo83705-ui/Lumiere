## 🚀 기획 및 설계 과정 회고 (Process & Retrospective)
### [박성은]
이번 프로젝트의 기획 단계에서는 단순한 아이디어 도출을 넘어, 실제 서비스 수준의 트래픽과 보안을 감안한 **기능적/비기능적 요구사항(NFR) 정의, 데이터 파이프라인 설계, 사용자 흐름(Flow) 구조화**에 집중했습니다.

### 1. 요구사항 정의 및 제약사항 도출
* [cite_start]**진행 내용:** 문제 상황(발색 왜곡, 정보 과부하)을 정의하고, 이를 해결하기 위한 기능(F-01~F-05)과 시스템이 갖춰야 할 비기능적 요구사항(NF-101~NF-403)을 명세서로 문서화했습니다. [cite: 12, 18, 46, 47, 48, 49, 50, 169, 174, 188]
* **어려웠던 점:** 처음에는 '빠르게 작동해야 한다', '보안이 좋아야 한다'처럼 추상적이었던 기준을 수치화하는 것이 까다로웠습니다.
* **새로 배운 것:** 비기능적 요구사항을 Pandas DataFrame으로 구조화해 보며, NFR은 구체적이고 측정 가능해야 한다는 것을 배웠습니다. [cite_start](예: 진단 응답 속도 최대 7초 이내 제한 [cite: 169], 얼굴 이미지 즉시 폐기(Volatile Storage) [cite: 174]).
* **느낀 점:** IT 인프라를 설계할 때는 단순히 성능을 높이는 것보다, 보수적인 관점에서의 '안정성'과 '개인정보 보호'가 시스템의 가장 중요한 기둥이 된다는 것을 다시 한번 체감했습니다.

### 2. 데이터 탐색 및 수집 파이프라인 기획
* **진행 내용:** 추천 시스템의 핵심이 될 데이터셋(AI Hub 안면 이미지, Makeup API, 쇼핑몰 크롤링)을 정의했습니다. 
* [cite_start]**어려웠던 점:** 외부 쇼핑몰 URL을 분석하여 화장품 호수와 색상 데이터를 가져오는 과정에서, 법적 문제와 서버 부하를 동시에 고려해야 했습니다. [cite: 203, 204]
* [cite_start]**새로 배운 것:** 크롤러 로직을 짤 때 단순히 HTML을 긁어오는 것을 넘어, 대상 사이트의 `robots.txt` 정책을 준수하고 [cite: 179] Request 간격을 조절하여 트래픽 과부하를 방지하는 보안 준수 사항의 중요성을 학습했습니다.

### 3. 다이어그램 설계 (Use Case & Flowchart)
* **진행 내용:** 시스템의 전체 숲을 보여주는 Use Case Diagram과 접속 순서를 명확히 하는 User Flow를 PlantUML을 활용해 코드로 시각화했습니다.
* **어려웠던 점:** 사용자의 접속 순서와 기능 나열을 혼동하여 다이어그램의 목적을 섞어서 생각했던 시행착오가 있었습니다.
* **새로 배운 것:** * `Use Case Diagram`: "누가 어떤 기능을 쓰는가"를 정의하여 시스템의 경계를 짓는 도구.
    * `Flowchart`: "어떤 조건 분기와 순서로 데이터가 흐르는가"를 정의하는 도구.
    * `<<include>>`와 `<<extend>>`를 활용해 예외 상황(예: 조명 인식 에러)을 모델링하는 법을 익혔습니다.
* [cite_start]**느낀 점:** 기획 초반에 이 두 가지 다이어그램을 분리해서 작성하니, 개발(Django 백엔드/FastAPI AI 서버 연동) 시 구현해야 할 API 엔드포인트와 화면의 개수가 훨씬 선명해졌습니다. [cite: 207, 208]

### 4. 화면 설계 (UI/UX) 및 WBS 수립
* **진행 내용:** NFR이 반영된 7개의 핵심 화면(로딩 중 Grad-CAM 시각화, 직관적 에러 모달 등) 구조를 짜고, Uizard 같은 AI 디자인 툴에 활용할 수 있는 프롬프트 형태로 와이어프레임을 기획했습니다. 이를 바탕으로 6주간의 WBS를 도출했습니다.
* [cite_start]**새로 배운 것:** 좋은 UI 기획은 "Error 500"을 띄우는 것이 아니라, "조명이 어두워요. 다시 찍어볼까요?" [cite: 184] 처럼 사용자 친화적인 행동 유도 메시지(UX Writing)까지 포함해야 함을 배웠습니다. 
* **느낀 점:** 백엔드 개발과 프론트 UI 퍼블리싱, 그리고 AI 모델(VQA, VLM, Grad-CAM 활용) 연동이라는 세 가지 큰 축을 6주 안에 완성하기 위해서는, API 명세서를 조기에 확정하고 병렬적으로 작업하는 것이 필수적이라는 전략을 세웠습니다.

### [김수진]

### 1단계: 문제 정의 (Problem Definition)
* **고민의 시작:** "왜 이 프로젝트가 필요한가?"라는 질문에 집중했습니다. 평소 관심 분야였던 뷰티 산업, 특히 올리브영 메이트로 근무하며 직접 목격했던 고객들의 어려움을 떠올렸습니다.
* **핵심 문제점:** 퍼스널 컬러 진단은 받았지만, 시중에 넘쳐나는 방대한 화장품 데이터 속에서 정작 자신에게 맞는 '호수'를 찾지 못해 구매를 망설이는 소비자들의 페인포인트를 발견했습니다. '손등 발색'의 한계를 AI 기술로 해결하고자 하는 명확한 이유를 정의할 수 있었습니다.

### 2단계: 목표 설정 (Goal Setting - SMART 원칙)
* **구체성 및 측정 가능성(Specific & Measurable):** 단순히 "화장품을 추천해 준다"가 아니라, 사용자의 피부색과 제품 색상의 유사도를 퍼센트(%) 수치로 산출하여 객관적인 근거를 제공하는 것을 목표로 설정했습니다.
* **현실적 달성 가능성(Achievable):** 모든 제품을 다루기보다 카테고리별 베스트셀러 데이터를 우선적으로 구축하고, 검증된 AI 라이브러리(MediaPipe 등)를 활용하여 프로젝트 기간 내에 구현 가능한 범위를 확정했습니다.
* **차별점:** 기존의 퍼스널 컬러 진단 서비스에서 한 발 더 나아가, 실질적인 제품 매칭 시스템 구축에 초점을 맞추어 서비스의 비전을 구체화했습니다.

### 3단계: 요구사항 분석 (Requirements Analysis)
* **기능적 요구사항:** 사용자가 가입 시 바로 퍼스널 컬러를 진단받는 프로세스부터, 특정 제품 선택 시의 추천 서비스와 능동적인 자동 추천 서비스까지, 사용자 입장에서 가장 편리한 기능을 리스트업했습니다.
* **비기능적 요구사항:** 사용자의 얼굴 사진(민감 정보)을 다루는 만큼 데이터 보안과 즉각적인 결과 반환을 위한 성능 최적화가 중요함을 깨닫고 이를 설계에 반영했습니다.
* **설계의 어려움:** 특히 ERD(Entity Relationship Diagram)를 구성하며 상품의 방대한 속성을 어떻게 구조화할지 고민하는 과정이 쉽지 않았으나, 이 과정이 서비스의 안정성을 결정짓는 핵심임을 학습했습니다.

### 4단계: 전체적인 느낀 점
* **사용자 중심 사고의 중요성:** 기획 단계에서 "사용자가 어떻게 하면 더 효율적이고 낭비 없는 쇼핑을 할 수 있을까?"를 끊임없이 질문하며 설계하는 과정이 매우 중요하다는 것을 느꼈습니다.
* **기술과 도메인의 결합:** 제가 가진 메이크업 분야의 경험을 AI 기술과 접목하는 과정에서 개발자로서의 시야가 넓어졌으며, 실제 서비스가 구현되었을 때 사용자가 느낄 효용성을 상상하며 큰 동기부여를 얻었습니다.
